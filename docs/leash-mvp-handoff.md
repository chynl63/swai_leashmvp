# Leash 웹 MVP — 구현·실험·분석 핸드오프 문서

> 이 문서는 **다른 Claude 세션에 그대로 입력해 "과제 결과 및 분석 방안"을 논의**하기 위한 컨텍스트 문서입니다.
> 작성 시점 기준으로 실제 구현된 내용만 담았습니다. (창업 수업 SWAI 과제용 검증 데모)

---

## 1. 프로젝트 개요

- **제품명**: Leash — "의지박약 인간을 위한 하드코어 스크린타임 앱"
- **핵심 컨셉**: 앱을 *막는* 것은 누구나 한다. Leash는 **차단을 *푸는* 것을 어렵게** 만든다. 차단(="폰의 산책")을 해제하려면 설정 시점에 정해진 **심리적 장벽(벌칙)**을 통과해야 한다.
- **이 MVP의 정체**: 실제 앱 차단은 iOS 네이티브(FamilyControls)에서만 가능하므로, **웹에서 핵심 플로우를 시뮬레이션**하는 인터랙티브 데모. 동시에 **사용자 행동을 로깅하는 검증 도구**.
- **기술 스택**: Next.js 15 (App Router) · TypeScript · Tailwind v4 · Zustand(localStorage persist) · Framer Motion
- **백엔드/DB**: **Google Apps Script + Google Sheets** (프런트에서 JSONP로 호출, CORS 회피). 웹앱 `/exec` URL은 코드(`lib/sheetdb.ts`)에 내장.
- **배포**: Netlify (GitHub 연동 자동 빌드). 레포: `chynl63/swai_leashmvp`
- **데모 가속**: 화면 우하단 ⚡토글. 기본 ON. 차단 타이머 60배(1시간=1분), 대기 벌칙 12배(3분→15초), 벤치 20배(5분→15초).

## 2. 검증 가설과 실험 설계 (가장 중요)

### 가설
> **마찰(벌칙)이 충분히 크면, 사람들은 차단 풀기를 포기한다.**

### 측정이 어려운 이유와 해결책 (인센티브 설계)
웹 데모에선 실제로 앱이 막히지 않으므로 "풀고 싶은 욕구"가 안 생긴다 → 그냥 클릭만 하고 끝나 데이터가 무의미해진다.
그래서 **실제 금전 인센티브(배달의민족 2만원권 추첨, 5명)**를 "지금 당장 앱을 쓰고 싶은 욕구"의 **대역(proxy)**으로 사용한다.

- 인센티브는 **"차단을 풀면(=벌칙을 끝까지 통과하면) 추첨 응모"**로 건다.
- 따라서 사람들은 상금을 보고 풀려는 **진짜 동기**가 생긴다.
- **그럼에도 벌칙이 귀찮아서 중간에 포기하면 → 마찰이 (금전 유인조차 이기는) 효과가 있다**는 강력한 증거가 된다.
- 이는 **보수적 하한 추정(lower bound)**: 일회성 추첨이 실제 도파민 갈망보다 약할 수 있으나, "금전 인센티브가 걸려 있어도 N%가 포기했다"는 결론은 방법론적으로 단단하다.

### 통제 변수 — 벌칙 고정
실험의 핵심 독립변수(마찰)는 **모든 참가자에게 동일하게 고정**한다(자기선택 교란 제거).
- **고정 벌칙 시퀀스** (`lib/profiles.ts`의 `FIXED_BARRIERS`): **① 수학 5문제 → ② 반성문 타이핑 → ③ 3분 대기**
- 사용자는 벌칙을 고를 수 없다(설정 화면엔 읽기 전용으로 표시). 앱 선택·차단 시간은 개인화로 남겨둠(마찰엔 영향 적음).
- 코드상 다른 벌칙(감시자 승인 `guardian`, 벌금 `fine`)도 존재하나 **현재 실험 시퀀스엔 미포함**.

### 측정의 핵심
- **벌칙 풀기를 시작했는가** → 풀었는가 / 중간에 포기했는가, 그리고 **왜**.
- **포기율(give-up rate)** = 마찰 효과의 직접 지표.
- 완료자에겐 **"상금 없었어도 풀었을까?"**(반사실)를 물어 인센티브 의존도를 보정.

## 3. 사용자 플로우

```
랜딩페이지(폰 목업 데모 임베드)
  └ "사용해보기"/스크롤 → 데모(iframe, /demo)
설정 화면: 묶을 앱 선택 + 차단 시간(◀▶) + [고정 벌칙 표시] → "🔗 목줄 채우기"
  → 산책 화면(/demo/walk): 타이머 + 걷는 캐릭터 + 발자국 + 응시 감지 + 가짜 유혹 푸시 알림
  → (홈 버튼=하단 바) → 스프링보드(/demo/springboard, iOS 홈)
  → 차단된 앱(🔒) 탭 → 앱 열리는 줌인 → 인터럽트(/demo/interrupt)
인터럽트 3버튼:
  ① "딱 5분만 사용하기"(벤치, 2h+ & 절반경과 시만 노출)
  ② "벌칙 풀고 차단 해제하기"(+추첨 응모 안내)  → 벌칙 시퀀스
  ③ "그냥 두고 산책 계속하기"(참기)            → resist 전환 → 산책 복귀
벌칙 시퀀스(수학→타이핑→대기), 각 화면 상단 "✕ 포기" 버튼:
  - 포기 → 포기 사유 설문(/demo/giveup) → resist 전환 → 산책 복귀
  - 끝까지 통과 → 완료자 설문+응모(/demo/entry) → 상처 전환(/demo/transition?kind=hurt) → 차단 해제 → 홈
통계 탭(/demo/stats): 주간 산책시간·연속기록·벌금·참음/줄끊음 히스토리
```

부가 화면: 감시자 승인 웹페이지(`/approve/[token]`, 프레임 밖), 랜딩 피드백 폼(이메일+한마디).

## 4. 구현된 기능 요약

- **랜딩**: 앱 팔레트(오프화이트+골든오커 #E8A835), 히어로(캐릭터), **실제 동작하는 폰 목업 데모를 iframe으로 임베드**, 폰 양옆에 화면별로 바뀌는 floating 안내 말풍선(`DemoTips`, iframe→부모 postMessage), 작동방식/장벽 소개, 추첨 안내 배지(펄스), 피드백 폼.
- **폰 목업**: 실제 iPhone 비율(390:844) 고정, 상태바/다이나믹아일랜드/홈버튼.
- **캐릭터(목줄 찬 폰)**: line-art SVG. 정면(인터럽트/응시) + 옆모습 2프레임(side-a/side-b) 반복 걷기 + 발자국 트레드밀(발에서 생겨 왼쪽으로 페이드). 이펙트: 반짝/빗금/눈물/홍조.
- **벌칙**: 수학(5문제 연속, 오답 흔들림) / 반성문 타이핑(문장 정확 일치) / 대기(화면 이탈 시 리셋). 각 상단 포기 버튼.
- **인센티브/응모**: 인터럽트·랜딩·완료화면에 "배민 2만원권 추첨 5명" 노출. 완료 시 이메일 응모.
- **설문**: 포기 사유 설문(/demo/giveup), 완료자 설문(/demo/entry).
- **데모 가속 토글**, **응시 감지**(30/90/180초), **유혹 푸시 알림**.

> 주의: **실제 추첨은 진행하지 않는 가상 인센티브**(예산 없음). 참가자에겐 진짜처럼 보이게 하되, 분석/발표 시 이 점을 명시하고 디브리핑하는 것을 권장(연구윤리).

## 5. 데이터 수집 — Google Sheets 스키마

스프레드시트에 4개 탭. **Apps Script `insert`는 `id` 열이 반드시 있어야 동작**(없으면 "id column is missing" 에러).

### `events` 탭 — 모든 행동 이벤트 + 설문 (분석의 핵심)
| 열 | 내용 |
|----|------|
| `id` | 6자 코드(원본 방식 `Math.random().toString(36).substring(2,8).toUpperCase()`). 이벤트마다 고유(행 식별용) |
| `type` | 이벤트 종류 (아래 6절 표 참고) |
| `detail` | 종류별 부가정보(문자열) |
| `created_at` | ISO 타임스탬프 |
→ **4열 고정. 새 열 추가 불필요.** 모든 이벤트가 `type`으로 구분되고 세부는 `detail`에 담긴다.

### `visitors` 탭 — 방문 로깅 (세션당 1회, 랜딩 진입 시)
| id | landingUrl | ip | referer | time_stamp | utm | device | email | advice |
→ **A1 헤더가 반드시 `id`여야 함**(과거 스키마라 `GN4LDM`이었으면 `id`로 변경해야 로깅됨). email/advice 열은 방문 로깅에선 비워둠.
→ **id = 방문자 UV**(원본 `getUVfromCookie` 동일): 쿠키 `user`(180일)에 저장해 **같은 사용자는 동일 id**. Apps Script가 같은 id면 업서트(행 갱신)하므로 **방문자 1명 = 1행**.
→ 주의: 이 탭엔 **예전 도파민락커 프로젝트의 레거시 데이터**가 섞여 있을 수 있음(landingUrl이 celadon-gelato/127.0.0.1). Leash 분석 시 분리 필요.

### `advice` 탭 — 랜딩 피드백 폼
| id | email | advice | created_at |

### `approval_requests` 탭 — 감시자 승인(현재 실험 벌칙엔 미사용, 기능만 존재)
| id | guardian_name | status | created_at |

> 응모 이메일은 `events` 탭에 `type=entry`로 들어간다(별도 `entries` 탭은 선택; 없어도 됨).

## 6. events 이벤트 종류 (type / detail)

| `type` | 언제 | `detail` 예시 | 의미 |
|--------|------|---------------|------|
| `session_start` | 목줄 채우기 | `집중 모드 · 120분 · 벌칙[math+typing+wait] · 앱[...]` | 차단 시작 |
| `session_completed` | 차단 시간 만료 완주 | `집중 모드` | 끝까지 산책 완료 |
| `leash_broken` | 벌칙 통과해 해제 | `집중 모드` | 차단 해제됨(줄 끊음) |
| `interrupt_reached` | 인터럽트 화면 진입 | (빈값) | **퍼널 시작점** — 풀고 싶어진 시점 |
| `unblock_start` | "벌칙 풀고 해제" 누름 | `math+typing+wait` | **해제 시도 시작** |
| `penalty_giveup` | 벌칙 중 "포기" 누름 | `math` (포기한 단계) | **중간 포기(존재 + 어느 벌칙)** |
| `giveup_reason` | 포기 사유 설문 제출 | `math \| 너무 귀찮아서` | **왜 포기했는지** |
| `unblock_complete` | 벌칙 전부 통과(완료 화면 도달) | (빈값) | **끝까지 풀어냄** |
| `complete_survey` | 완료자 설문 제출 | `상금없어도=아니오 \| 귀찮음=4` | **완료자 반사실 + 마찰 강도** |
| `entry` | 응모(이메일 제출) | `user@email.com` | 추첨 응모 |
| `resist` | "그냥 두기" 또는 포기 후 | `집중 모드` | 차단 유지 선택(=해제 안 함) |
| `bench` | "딱 5분만" 진입 | `인스타그램` | 5분 임시 사용 |
| `fine` | (현재 미사용 벌칙) | `₩2,250` | 벌금 결제 |

**포기 사유 설문 보기**: 너무 귀찮아서 / 시간이 오래 걸려서 / 벌칙이 어려워서 / 그 앱 그렇게 급하진 않아서 / 그냥 안 쓰기로 / 기타(직접입력) / 무응답
**완료자 설문**: Q1 "상금 없었어도 끝까지 풀었을까?"(네 / 아니오, 상금 때문 / 모르겠음), Q2 "얼마나 귀찮았나"(1~5)

## 7. 분석 방안

### 7.1 핵심 지표
- **포기율 (friction effectiveness)** ≈ `count(penalty_giveup) / count(unblock_start)`
  - 또는 `1 − count(unblock_complete)/count(unblock_start)` (포기 버튼 없이 이탈한 경우까지 포함)
  - **해석**: 금전 인센티브가 걸려 있는데도 이 비율만큼 포기 → 마찰이 효과 있다(하한).
- **완료율** ≈ `count(unblock_complete) / count(unblock_start)`
- **인센티브 의존도(완료자 중)**: `complete_survey`에서 `상금없어도=아니오` 비율 → "현실(상금 없음)이면 포기했을 사람" = 효과의 추가 근거.

### 7.2 퍼널 (집계 카운트)
```
visitors(방문) → session_start(차단 시작) → interrupt_reached(풀고 싶어짐)
  → unblock_start(시도) → ┬ penalty_giveup → giveup_reason   (포기)
                          └ unblock_complete → complete_survey → entry  (완료)
또한 interrupt에서 resist(그냥 두기) = 시도조차 안 함
```
각 단계 카운트는 `events`에서 `type`별 `COUNTIF`로 산출.

### 7.3 교차/세부 분석
- **단계별 포기 분포**: `penalty_giveup`의 `detail`(math/typing/wait) 빈도 → **어떤 마찰에서 가장 많이 포기**하는가 = 가장 효과적인 벌칙.
- **포기 사유 분포**: `giveup_reason`의 detail 파싱 → 귀찮음 vs 어려움 vs 불필요 등.
- **마찰 강도 체감**: `complete_survey`의 귀찮음(1~5) 분포 + 포기자와 비교.
- **유입 경로**: `visitors`의 `utm`/`referer`/`device`별 포기율 차이.
- **참여/이탈**: 방문 대비 session_start 전환율, interrupt 대비 resist(그냥 두기) 비율.

### 7.4 시트에서 계산 예
- 포기 수: `=COUNTIF(events!B:B, "penalty_giveup")`
- 시도 수: `=COUNTIF(events!B:B, "unblock_start")`
- 포기율: 위 둘의 비
- 사유 집계: `giveup_reason` 행만 필터 후 detail의 `|` 뒤 텍스트 집계

## 8. 한계 및 주의 (발표/리포트에 명시 권장)

1. **인센티브 ≠ 실제 갈망**: 일회성 추첨은 습관적·충동적 도파민 갈망과 성질이 다름 → 결과는 **하한 추정**으로 해석.
2. **추첨 확률 체감**: 상금 기대값이 작으면(당첨 인원 적음/응모 많음) 쉽게 포기 → 포기율이 과장될 수 있음. "5명 추첨"의 체감 확률이 결론 강도를 좌우.
3. **데모 가속**: 기본 ON이면 "3분 대기"가 15초로 줄어 마찰이 약화됨. **실제 수집 시 ⚡가속 OFF 권장**(또는 대기 단계 해석 시 주의).
4. **이벤트에 세션/유저 ID 없음**: `events` 행은 개별 이벤트라 한 사람의 여정을 완벽히 잇기 어려움 → **집계 카운트 기반 분석이 안전**. (개선: 세션 ID 컬럼 추가하면 사용자 단위 퍼널 가능)
5. **조용한 이탈**: 포기 버튼 없이 탭 닫기/홈버튼으로 나간 사람은 `giveup_reason`이 없음 → `unblock_start`만 있고 `unblock_complete` 없는 것으로 추정.
6. **표본/자기선택**: 데모 방문자는 일반 사용자와 다를 수 있음(자원자 편향). 표본 크기 확인 필요.
7. **가상 추첨(deception)**: 실제 추첨 미실시 → 윤리상 디브리핑/고지 권장. 또한 데이터 정직성(상금 노린 무성의 통과) 가능성.
8. **localStorage 영속**: 재방문자는 이전 상태/통계가 남아 중복·잔여 데이터 가능.

## 9. 함께 논의하고 싶은 것 (다음 세션용)

- 위 데이터로 **"이 제약이면 사람들이 차단 풀기를 포기한다"**를 어떻게 가장 설득력 있게 주장할지(지표·시각화·서술).
- 표본·확률 체감·가속 등 한계를 보완할 **추가 측정/문항**이 필요한지.
- 단계별 포기 데이터로 **어떤 벌칙 조합이 최적**인지 도출하는 방법.
- 발표용 핵심 그래프(퍼널, 포기율, 사유 분포, 반사실 응답) 구성.
